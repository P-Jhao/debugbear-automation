# DebugBear 批量性能测试工具需求文档（简版）

## 1. 背景与问题
- 当前线上性能测试依赖人工在 DebugBear 网站逐次点击发起测试。
- 每个页面/场景需要重复测试约 12 次，过程耗时且易出错。
- 结果统计需要手动去极值、计算平均值，效率低且标准不统一。

## 2. 建设目标
- 提供一个简单前端工具，支持“一键发起 N 次性能测试”。
- 自动汇总每次测试结果，支持去极值后计算平均值。
- 在工具内展示关键指标，同时可跳转 DebugBear 官网查看单次测试详情。
- 支持按“版本号 -> 分组”管理测试任务，并与基线任务做指标对比。
- 降低人工操作成本，提高测试效率与数据一致性。

## 3. 适用范围
- 面向内部性能测试同学/研发同学使用。
- 本期优先支持通过 DebugBear API 对指定 URL 批量发起测试并统计结果。

## 4. 用户角色
- 测试执行人：配置测试参数并发起批量测试。
- 结果查看人：查看汇总指标，定位单次异常数据并跳转官网详情。

## 5. 核心流程
1. 用户输入测试 URL、测试次数（默认 12 次）及必要参数（如设备类型/地域等），并先选择版本号，再选择该版本下的分组。
2. 用户可选对比基线：指定“版本+分组”，指定版本（同组），或“上一次同 URL 同版本同分组测试任务”。
3. 点击“开始测试”，前端调用后端接口，由后端串行或限流并发调用 DebugBear API。
4. 系统实时展示进度（如：已完成 5/12）。
5. 全部完成后，系统按规则计算：
   - 原始平均值
   - 去极值平均值（去掉最高值和最低值后求平均）
6. 若配置了基线，系统输出当前任务与基线任务的指标变化（绝对值、百分比、升降趋势）。
7. 用户可点击每一条测试记录跳转到 DebugBear 官网详情页。
8. 支持导出结果（CSV/JSON，二期可选）。

## 6. 功能需求

### 6.1 参数配置
- 必填：测试 URL。
- 必填：测试次数（默认 12，范围建议 3~30）。
- 必填：`version`（版本号）。
- 必填：`group`（分组名，且归属所选 `version`）。
- 交互规则：先选版本，再加载并选择该版本下可用分组。
- 选填：测试环境参数（设备、地区、网络类型，按 DebugBear API 能力开放）。
- 选填：任务名称/备注，方便检索历史记录。
- 选填：对比基线（`baselineType`：`version_group`/`version`/`previous`）。

### 6.2 任务执行
- 支持创建一次批量测试任务。
- 支持任务状态：待开始、进行中、已完成、部分失败、失败。
- 接口错误自动重试（如单次请求失败最多重试 1~2 次）。
- 并发策略可配置（默认小并发，避免触发 API 限流）。

### 6.3 结果展示与统计
- 列表展示每次测试核心指标（如 LCP、INP、CLS、TTFB 等）。
- 自动计算每项指标：
  - 总平均值
  - 去极值平均值（默认去 1 高 + 1 低）
  - 最大值/最小值
- 标记异常值（超阈值高亮）。
- 每条记录附带 DebugBear 详情链接。

### 6.4 历史记录
- 可查看历史任务列表。
- 可按 URL、日期、任务状态、版本号、分组筛选。
- 支持重新运行同参数任务（可选）。

### 6.5 版本/分组对比
- 支持将当前任务与以下任一基线进行对比：
  - 指定版本 + 指定分组
  - 指定版本（同分组）
  - 上一次同 URL 同版本同分组测试任务
- 对比维度包含：LCP、INP、CLS、TTFB（后续可扩展）。
- 每项指标展示：
  - 当前值
  - 基线值
  - 变化值（`current - baseline`）
  - 变化比例（`(current - baseline) / baseline`）
  - 趋势（改善/变差/持平，按指标方向判定）
- 支持“按原始平均值对比”与“按去极值平均值对比”（默认去极值平均值）。

## 7. 非功能需求
- 易用性：3 步内可完成任务发起。
- 性能：12 次任务在 API 正常情况下应稳定完成并可见实时进度。
- 可维护性：统计规则可配置（如去极值数量可改为去 2 高 2 低）。
- 安全性：
  - DebugBear API Key 不在前端暴露，统一由后端安全存储和调用。
  - 日志中对密钥进行脱敏。

## 8. 接口与数据（建议）

### 8.1 后端接口（示例）
- `POST /api/perf-tasks`
  - 入参：`url`、`count`、`config`、`version`、`group`、`baselineType?`、`baselineTarget?`
  - 出参：`taskId`
- `GET /api/versions`
  - 出参：版本列表
- `GET /api/versions/{version}/groups`
  - 出参：该版本下的分组列表
- `GET /api/perf-tasks/{taskId}`
  - 出参：任务状态、进度、每次测试结果、统计结果
- `GET /api/perf-tasks`
  - 出参：历史任务列表
- `GET /api/perf-tasks/{taskId}/compare`
  - 出参：基线任务信息、各指标对比结果

### 8.2 结果字段（示例）
- 任务字段：`taskId`、`url`、`version`、`group`、`baselineType?`、`baselineTarget?`、`createdAt`
- 单次结果：`runId`、`debugBearUrl`、`lcp`、`inp`、`cls`、`ttfb`、`status`、`createdAt`
- 汇总结果：`avg`、`trimmedAvg`、`max`、`min`、`successCount`、`failCount`
- 对比结果：`metric`、`currentValue`、`baselineValue`、`diff`、`diffRate`、`trend`

## 9. 统计规则定义（V1）
- 当成功样本数 `< 3`：不计算去极值平均值，仅展示原始平均值。
- 当成功样本数 `>= 3`：去掉每项指标的最高值和最低值各 1 个后求平均。
- 失败样本不参与平均值计算，但需计入失败次数。

## 10. 验收标准（V1）
- 能对任意 URL 一键发起 12 次测试并完成。
- 页面可实时显示任务进度和每次结果。
- 自动输出原始平均值与去极值平均值，计算结果正确。
- 能在任务创建时先选择版本，再选择该版本下分组，并可用于历史筛选。
- 能选择基线（指定版本+分组/指定版本同组/上一次同版本同分组）并正确展示指标变化值与变化比例。
- 可点击跳转到 DebugBear 对应单次详情页。
- API Key 不出现在浏览器网络请求明文中。

## 11. 里程碑建议
1. M1（1~2 天）：完成后端 API 打通与最小可用批量执行。
2. M2（1~2 天）：完成前端任务创建、进度展示、结果列表。
3. M3（1 天）：补齐统计规则、异常处理、历史记录与验收测试。

## 12. 风险与待确认项
- DebugBear API 的调用频率限制与计费规则需提前确认。
- 指标口径需和团队统一（是否按移动端优先、是否固定地域）。
- “上一次测试组”定义需统一（按同 URL + 同环境参数，还是仅同 URL）。
- 对比趋势判定规则需统一（例如 LCP 越小越好，CLS 越小越好）。
- 版本与分组的主数据来源需统一（手动维护、从发布系统同步，或自动创建）。
- 是否需要对外导出报表（CSV）作为版本/分组对比基线。
